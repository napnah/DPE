# DPE 项目说明

## 1. 项目是干什么的

DPE，全称 Distributed Privacy Editor，是一个分布式隐私协作文档编辑器。它想解决的问题是：多个人在同一个局域网或可信网络环境里协作文档时，既能像在线文档一样实时同步，又不把文档内容完全交给中心服务器保存和转发。

项目的核心目标可以概括为三点：

- 支持多人协同编辑文档。
- 文档更新在 P2P 网络中传输，内容以加密二进制形式同步。
- 对每个文档节点设置细粒度权限，控制成员是否可见、可读、可写、可管理。

所以它不是一个普通的单机 Markdown 编辑器，也不是传统的中心化网盘文档系统。它更像是“带权限控制和加密同步能力的局域网协作文档系统”。

## 2. 用户能做什么

从使用流程看，用户主要会经历这些步骤：

1. 打开 Web 页面，浏览器本地生成一个节点身份。
2. 创建群组，成为群组拥有者。
3. 邀请其他成员加入群组。
4. 在群组根目录下创建子文档。
5. 给不同成员或角色配置文档权限。
6. 多个成员进入同一文档后进行协作编辑。
7. 系统通过 WebRTC、Yjs 和加密协议同步编辑内容。

项目中的根节点 `root` 是目录，不是直接编辑的正文文档。实际编辑通常发生在根目录下面创建的子文档里。

## 3. 权限模型

DPE 的权限按文档节点划分，每个 `doc_id` 都有自己的权限和密钥。父目录可读不代表子文档一定可读，子文档是否可见和可解密取决于它自己的 ACL 与 RBAC 展开结果。

权限分为 4 级：

| 级别 | 名称 | 能力 |
|------|------|------|
| 0 | Invisible | 不可见，不可解密，不参与同步 |
| 1 | Readonly | 可读取和合并别人已授权的更新，但不能产生会被接受的写更新 |
| 2 | Writable | 可读写文档内容 |
| 3 | Operable | 在可写基础上，还能执行创建子文档、删除文档、设置 ACL 等管理操作 |

群组里还有角色体系，例如管理员、协作者、读者。角色主要用于批量授权和默认规则：新成员默认是什么角色，新建文档默认给不同角色什么权限。

## 4. 系统架构

项目把系统拆成两个逻辑平面：

| 平面 | 作用 |
|------|------|
| 控制平面 | 负责群组、邀请、文档树、权限、JWT 签发、密钥生命周期和管理 RPC |
| 数据平面 | 负责文档内容的 P2P 同步、Yjs CRDT 合并、加密更新传输和合并前校验 |

控制平面不是高频同步通道。用户打字产生的文档更新主要通过数据平面在节点之间传输。控制平面只在发证、改权限、创建文档、轮换密钥等结构性操作时介入。

## 5. 关键技术

项目主要使用这些技术：

| 技术 | 用途 |
|------|------|
| React + Vite | 前端界面 |
| Yjs | 协同编辑 CRDT，同步多人编辑状态 |
| WebRTC DataChannel | 节点之间的 P2P 数据传输 |
| mDNS / LAN discovery | 局域网节点发现 |
| NestJS | 控制平面后端服务 |
| PostgreSQL | 存储群组、成员、文档树、ACL、角色等控制平面数据 |
| Prisma | 数据库 schema 和访问层 |
| Ed25519 | 节点身份、签名、JWT 签发 |
| AES-GCM | 加密 Yjs 文档更新 |
| pnpm + Turbo | Monorepo 包管理和多包任务编排 |

## 6. 安全设计思路

DPE 的安全设计重点不是“所有节点都完全可信”，而是通过密码学和合并策略限制未授权成员。

核心机制包括：

- 每个浏览器节点生成 Ed25519 密钥对，`NodeID` 由公钥哈希派生。
- 每个文档节点有独立的对称密钥 `Key(doc)`。
- 只有权限大于等于 1 的用户，控制平面才会给他签发包含该文档密钥密文的 JWT。
- Yjs 原始 update 不直接明文发送，而是封装为 `SignedUpdate`。
- `SignedUpdate` 包含 `doc_id`、`key_version`、nonce、密文、发送者、序号和签名。
- 接收方在 `Y.applyUpdate` 前必须检查 JWT、权限级别、签名、密钥版本和重放序号。
- 只读用户即使发出写更新，其他节点也应该在合并前拒绝。

也就是说，项目不是只靠 UI 禁用按钮来限制权限，而是在同步协议和合并入口处做强制校验。

## 7. 仓库结构

主要目录如下：

```text
apps/
  web/              React 前端页面
  control-plane/    NestJS 控制平面，处理群组、邀请、ACL、JWT、文档树
  signaling/        WebRTC 信令服务
  lan-agent/        局域网发现与邻居探测

packages/
  crypto/           加密、签名、密钥封装、JWT 等基础能力
  proto/            协议 schema 和共享协议类型
  acl/              权限策略判断
  p2p/              P2P 握手、信令协议、通道封装
  yjs-provider/     安全 Yjs provider，处理加密 update 和合并前校验
  shared/           通用工具和共享类型

docs/               架构、设计、阶段验收文档
todo/               开发模块划分、测试计划和项目说明
scripts/            构建、验收、安全审计、E2E 脚本
```

## 8. 本地运行方式

项目要求：

- Node.js 20+
- pnpm 9+
- Docker，用于本地启动 PostgreSQL

首次初始化：

```bash
pnpm install
cp .env.example .env
docker compose up -d postgres
cd apps/control-plane && pnpm db:push && cd ../..
```

日常启动：

```bash
pnpm dev
```

启动后访问：

```text
http://localhost:5173
```

各服务端口：

| 服务 | 端口 | 地址 |
|------|------|------|
| Web | 5173 | http://localhost:5173 |
| Control plane | 3001 | http://localhost:3001/health |
| Signaling | 3002 | http://localhost:3002/health |
| LAN agent | 3003 | http://localhost:3003/health |

## 9. 这个项目的重点和难点

项目的重点不只是“做一个文档编辑器”，而是把协同编辑、P2P 网络、权限控制和加密协议组合起来。

主要难点包括：

- 协同编辑更新必须能实时合并，不能因为网络顺序不同导致冲突。
- 文档内容不能在 P2P 传输中明文暴露。
- 权限不能只在前端判断，必须在协议层和合并层生效。
- 用户权限变化后，需要考虑密钥轮换和旧 token 失效。
- P2P 连接建立前需要身份握手，防止未授权节点加入同步。
- 控制平面和数据平面要分工清楚，避免高频编辑流量都经过中心服务。

## 10. 一句话总结

DPE 是一个面向局域网和可信协作场景的加密 P2P 协作文档系统：用控制平面管理身份、群组和权限，用数据平面同步加密后的 Yjs 更新，在保证协作体验的同时尽量减少中心化服务对文档内容的介入。
