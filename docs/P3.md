# P3 — Data Plane Bootstrap (Signaling, Discovery, AuthEnvelope)

## Scope

| Component | Deliverable |
|-----------|-------------|
| `@dpe/p2p` | AuthEnvelope validation + first-message handshake; WebRTC signaling message schemas; `WireFrame` schema for post-handshake frames |
| `apps/signaling` | Room-based WebSocket relay (`join` / `leave` / `signal` → `peers` / `signal` / `error`) |
| `apps/lan-agent` | mDNS (`bonjour-service`) + manual peers + probe fallback + UID search + discovery WebSocket |

Covers plan **总-2 / 总-3** (network info, LAN neighbors, UID search). Full browser WebRTC mesh and Yjs sync → **P4**.

## Protocol

### Signaling (`ws://host:3002/ws`)

Client → server:

```jsonc
{ "type": "join",   "room": "<group_id>", "node_id": "<NodeID>" }
{ "type": "leave",  "room": "<group_id>" }
{ "type": "signal", "room": "<group_id>", "to": "<optional NodeID>", "payload": { ... } }
```

Server → client:

```jsonc
{ "type": "peers",  "room": "<group_id>", "peers": ["..."] }
{ "type": "signal", "room": "<group_id>", "to": "...", "payload": { "from": "<senderNodeID>", ... } }
{ "type": "error",  "code": "invalid_message" | "not_joined", "message": "..." }
```

Notes:

- The server **fills in** `payload.from = client.nodeId` on relay; clients should not set it themselves.
- A client that re-`join`s with a different `node_id` is treated as a re-identification and previously joined rooms are vacated (see `apps/signaling/src/rooms.ts`).
- Signaling is **unauthenticated** by design — authorization happens on the DataChannel first frame (see below).

### AuthEnvelope (DataChannel first frame)

After WebRTC connects, the initiator sends `AuthEnvelope` (see `packages/proto`). The responder runs `acceptAuthEnvelope()` from `@dpe/p2p`, which:

1. Awaits the first DataChannel message (default timeout **15 s** — configurable via `timeoutMs`).
2. Parses it against `authEnvelopeSchema`; any subsequent frame arriving before settlement rejects with `not_first_message`.
3. Verifies the JWT with the pinned `pk_admin` (`audience` = `group_id`, optional `issuer`).
4. Checks `payload.sub === envelope.node_id` (prevents JWT-to-NodeID swap).
5. If a `challenge` is configured, verifies `proof` against the peer's public key — this path **requires** `validateAuthEnvelopeWithPeerKey` (the plain `validateAuthEnvelope` rejects if a challenge is configured without a peer key).

On any failure the channel is closed and the promise rejects with `AuthHandshakeError` whose `reason` is one of:

| `reason` | Meaning |
|----------|---------|
| `timeout` | No first frame within `timeoutMs` |
| `invalid_envelope` | First frame failed schema parse / not JSON |
| `jwt_invalid` | JWT signature / audience / issuer / exp check failed |
| `node_mismatch` | `payload.sub` does not match `envelope.node_id` |
| `proof_invalid` | challenge proof missing or signature invalid |
| `not_first_message` | A subsequent frame arrived before handshake settled |

The responder does **not** send an error frame back to the initiator on failure — it simply closes the channel. The initiator detects rejection via `onclose`.

### Post-handshake frames (`WireFrame`)

After AuthEnvelope settles, the channel carries application frames defined by `wireFrameSchema` in `@dpe/p2p`:

```ts
type WireFrame =
  | { type: "auth";    payload: Record<string, unknown> }   // future re-handshake
  | { type: "sync";    payload: Uint8Array | string }       // Yjs SignedUpdate (P4)
  | { type: "control"; payload: Record<string, unknown> };  // KeyRotation / ACL notifications
```

`@dpe/p2p` provides the **schema only**; routing and consumption belong to `@dpe/yjs-provider` (P4).

### LAN agent

| Method | Path | Description |
|--------|------|-------------|
| GET | `/` | Self-describe (service, endpoints, hints) |
| GET | `/health` | `{ status, service, node_id, peer_count }` |
| GET | `/network` | Hostname, node_id, interfaces, signaling URL |
| GET | `/discovery` | Current peer list |
| GET | `/peers?uid=` | Filter peers by NodeID substring (case-insensitive) |
| POST | `/peers/manual` | Register manual peer `{ uid, host, port, name? }` |
| WS | `/ws` | Server pushes `{type:"discovery", peers}` on changes; client `{type:"ping"}` → server `{type:"pong", node_id}` |

Each discovered peer carries a `source` field:

```ts
type LanPeer = {
  uid: string;
  host: string;
  port: number;
  name?: string;
  source: "mdns" | "manual" | "probe";
  lastSeen: number;
};
```

Source precedence: **`manual` > `mdns` / `probe`** — a manual peer is never overwritten by mDNS/probe updates (`apps/lan-agent/src/discovery.ts upsert()`). The mDNS service type is `"dpe-agent"` (`@dpe/p2p.DPE_MDNS_SERVICE_TYPE`, kept stable for backward compatibility). CORS is enabled for all origins so browsers on `http://<lan-ip>:5173` can call the REST endpoints.

## Env

| Variable | Purpose |
|----------|---------|
| `SIGNALING_PORT` | signaling listen port (default `3002`; does **not** fall back to `PORT`) |
| `LAN_AGENT_PORT` | lan-agent listen port (default `3003`) |
| `DPE_NODE_ID` | this agent's NodeID (UID); defaults to `local-<hostname>` |
| `DPE_AGENT_NAME` | mDNS display name (defaults to hostname) |
| `DPE_AGENT_HOST` | self-advertised LAN host; auto-picked from network interfaces if absent |
| `DPE_LAN_HOST` | secondary hint for `pickLanAddress()` |
| `DPE_MDNS_INTERFACE` | bind mDNS to a specific NIC (multi-NIC machines) |
| `DPE_DISABLE_MDNS=1` | skip Bonjour entirely (CI / restricted networks) |
| `DPE_MANUAL_PEERS` | `uid@host:port,...` to seed peers at startup |
| `DPE_DISCOVERY_PROBE_HOSTS` | comma-separated peer IPs to poll `/network` — **required for dual-host when mDNS is one-way** (NAT virtual NICs); polled every 15 s |
| `DPE_SIGNALING_URL` | advertised to clients via `/network.signalingUrl` (lan-agent does not connect to it itself) |

## Verify

```bash
pnpm verify:p3            # build + unit tests (offline)
pnpm verify:p3 --live     # + signaling + lan-agent WS/REST smoke
```

The `--live` mode spawns signaling on `127.0.0.1:3098` and lan-agent on `127.0.0.1:3097` (with `DPE_DISABLE_MDNS=1`), deliberately offset from the dev ports (`3002`/`3003`) so it can coexist with a running `pnpm dev`.
