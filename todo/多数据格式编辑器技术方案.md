# 多数据格式编辑器技术方案

## 1. 背景

当前 DPE 前端文档编辑能力集中在 `apps/web/src/components/DocInlineEditor.tsx`。它的实现方式是：

- 每个文档创建一个 `Y.Doc`。
- 正文内容固定使用 `doc.getText("content")`。
- UI 使用一个原生 `textarea` 展示和编辑纯文本。
- 本地持久化使用 `Y.encodeStateAsUpdate(doc)` 保存完整 Yjs 状态到 `localStorage`。
- 远端快照通过 `state_update_base64` 保存完整 Yjs 状态。
- P2P 同步通过 `SecureYjsProvider` 广播 Yjs update，底层会加密、签名并做合并前权限校验。

因此，目前“只能编辑纯文本”主要是 UI 层和 Yjs 数据建模较简单造成的，不是同步协议只能传纯文本。只要新的内容仍写入同一个 `Y.Doc`，现有加密同步、快照持久化、权限校验都可以复用。

## 2. 目标

为演示场景扩展文档可表达的数据类型，优先支持：

- 纯文本：继续兼容现有文档。
- Markdown：支持 Markdown 编辑和预览。
- 图片：支持插入本地图片并在文档中展示。
- 结构化块：为后续表格、代码块、任务列表、附件卡片等格式预留扩展点。

本次设计目标是“演示效果明显、改动边界可控、兼容现有同步协议”。

## 3. 非目标

第一阶段不建议做以下能力：

- 不做完整富文本编辑器内核。
- 不做大文件网盘或对象存储。
- 不做图片的跨文档复用、 CDN、缩略图服务。
- 不改变 P2P SignedUpdate、JWT、ACL、密钥轮换等底层安全协议。
- 不强制修改后端数据库结构，除非后续要做大附件。

## 4. 当前代码约束

### 4.1 前端约束

`DocInlineEditor` 当前把 `Y.Text("content")` 和 `textarea` 直接绑定。代码中有两个关键点：

- `applyTextareaDeltaToYText(ytext, nextValue)` 只处理字符串差量。
- `wireTextarea()` 只管理一个文本输入控件。

如果直接在这个组件里叠加图片、Markdown 预览、块编辑，会让组件膨胀。更合适的做法是拆分：

- `DocInlineEditor`：保留文档会话、JWT、Y.Doc、provider、持久化的外壳。
- `PlainTextSurface`：原来的 textarea 纯文本编辑器。
- `MarkdownSurface`：Markdown 编辑与预览。
- `BlockDocumentSurface`：图片、代码块、表格等块结构编辑器。

### 4.2 数据约束

后端快照接口保存的是完整 Yjs state：

```text
state_update_base64
```

`DocStateService` 中快照大小限制是 2MB。图片如果直接以 base64 写入 Yjs，容易超过限制，并且会让 P2P update 变大。因此图片支持需要分级：

- 演示优先：限制图片大小，压缩为 Data URL 后存入 Yjs。
- 长期方案：新增附件存储接口，只在 Yjs 中保存附件元数据和引用。

### 4.3 文档树约束

当前 `DocNodeRow` 只有：

```ts
{
  docId,
  parentDocId,
  title,
  keyVersion,
  isFolder?
}
```

没有 `docType` 字段。因此第一阶段不在目录树层区分“Markdown 文档 / 图片文档 / 表格文档”，而是在文档内容内部记录模式。

## 5. 推荐数据模型

继续使用同一个 `Y.Doc`，新增几个 Yjs 类型：

```ts
const ytext = doc.getText("content");
const ymeta = doc.getMap("meta");
const yblocks = doc.getArray("blocks");
const yassets = doc.getMap("assets");
```

### 5.1 meta

`meta` 保存文档编辑模式和版本：

```ts
{
  schemaVersion: 1,
  editorMode: "plain" | "markdown" | "blocks",
  createdAt?: string,
  updatedAt?: string
}
```

默认兼容策略：

- 老文档没有 `meta.editorMode` 时，按 `plain` 处理。
- 已有 `content` 文本继续正常显示。

### 5.2 content

`content` 继续保留为主文本字段：

- `plain` 模式：纯文本。
- `markdown` 模式：Markdown 源码。
- `blocks` 模式：可以作为导出纯文本或 fallback 内容。

保留 `content` 的好处是兼容现有测试、快照和旧文档。

### 5.3 blocks

`blocks` 用于结构化内容，每个元素是一个普通 JSON 对象：

```ts
type DocBlock =
  | { id: string; type: "paragraph"; text: string }
  | { id: string; type: "heading"; level: 1 | 2 | 3; text: string }
  | { id: string; type: "markdown"; source: string }
  | { id: string; type: "image"; assetId: string; alt?: string; caption?: string }
  | { id: string; type: "code"; language?: string; code: string }
  | { id: string; type: "todo"; checked: boolean; text: string };
```

第一阶段可以只落地：

- `paragraph`
- `markdown`
- `image`
- `code`
- `todo`

表格可以后续扩展：

```ts
{ id: string; type: "table"; rows: string[][] }
```

### 5.4 assets

`assets` 用于保存图片元数据。演示阶段可以存 Data URL：

```ts
type ImageAsset = {
  id: string;
  kind: "image";
  name: string;
  mime: "image/png" | "image/jpeg" | "image/webp" | "image/gif";
  size: number;
  width?: number;
  height?: number;
  dataUrl: string;
  createdAt: string;
};
```

必须限制：

- 单张图片建议压缩到 256KB 以下。
- 单文档图片总量建议限制在 1MB 左右。
- 超限时前端直接拒绝插入并提示用户。

这样能避免超过后端 2MB snapshot 限制，也能避免 P2P 同步卡顿。

## 6. UI 设计

### 6.1 编辑器外壳

把 `DocInlineEditor` 调整为“文档会话容器”，职责包括：

- 加载身份和 JWT。
- 创建并恢复 `Y.Doc`。
- 创建 `SecureYjsProvider`。
- 注册 mesh provider。
- 管理本地和远端 snapshot 保存。
- 根据 `meta.editorMode` 渲染不同编辑 surface。

它不再直接渲染 textarea。

### 6.2 顶部工具栏

在文档区域顶部增加紧凑工具栏：

- 模式切换：纯文本 / Markdown / 块文档。
- 插入：图片、代码块、任务项。
- Markdown 视图：编辑 / 预览 / 分栏。
- 保存状态和只读状态展示。

权限规则：

- `role < 2` 时，工具栏插入和编辑按钮 disabled。
- 只读用户仍可切换预览模式，但不能修改内容。

### 6.3 Markdown 模式

Markdown 模式第一阶段建议使用“双栏或标签页”：

- 左侧或编辑 tab：textarea 编辑 Markdown 源码。
- 右侧或预览 tab：渲染 Markdown。

依赖选择：

- 轻量方案：自写一个最小 Markdown renderer，只支持标题、粗体、斜体、列表、代码块、链接、图片语法。优点是少依赖，缺点是功能有限。
- 推荐方案：引入 `react-markdown`，必要时配合 `remark-gfm`。优点是稳定、演示效果好，支持表格、任务列表等 GFM 能力。

考虑演示目标，推荐使用：

```bash
pnpm --filter @dpe/web add react-markdown remark-gfm
```

如果后续涉及 HTML 渲染，默认不要开启原始 HTML，避免 XSS 风险。

### 6.4 图片插入

图片插入流程：

1. 用户点击“插入图片”。
2. 选择本地图片文件。
3. 前端校验 MIME 和大小。
4. 使用 `createImageBitmap` 或 `canvas` 压缩到限制尺寸。
5. 生成 `assetId`，写入 `yassets`。
6. 在 `yblocks` 中插入 image block。
7. Yjs update 走现有 provider 同步。

支持格式：

- PNG
- JPEG
- WebP
- GIF 可先允许但不压缩，建议限制更小。

图片展示：

- 卡片式展示图片预览。
- 支持编辑 alt 和 caption。
- 支持删除图片块。

注意：删除图片块时，如果没有其他 block 引用该 asset，可以同步删除 `yassets[assetId]`。

### 6.5 块文档模式

块文档模式适合演示多数据格式混排：

- 段落块：普通文本。
- Markdown 块：小块 Markdown 源码和预览。
- 图片块：图片和说明。
- 代码块：语言选择和代码 textarea。
- Todo 块：checkbox + 文本。

第一阶段可以先做“块列表 + 每块内简单控件”，不需要复杂拖拽。

块操作：

- 添加块。
- 删除块。
- 上移 / 下移。
- 编辑块内容。

块顺序由 `Y.Array("blocks")` 决定。

## 7. 同步与持久化影响

### 7.1 P2P 同步

不需要修改 `SecureYjsProvider`。原因：

- provider 监听的是整个 `Y.Doc` 的 update。
- 不关心 update 来自 `Y.Text`、`Y.Map` 还是 `Y.Array`。
- 加密、签名、防重放、权限校验仍然作用于完整 update。

### 7.2 本地持久化

不需要修改 `doc-persistence.ts` 的核心逻辑。它保存完整 Yjs state，可以自然包含 `meta`、`blocks`、`assets`。

建议新增测试覆盖：

- 保存并恢复 `Y.Map("meta")`。
- 保存并恢复 `Y.Array("blocks")`。
- 保存并恢复 `Y.Map("assets")` 中的小图片 Data URL。

### 7.3 远端快照

不需要修改接口结构，但需要前端控制 snapshot 大小。

建议在保存前计算：

```ts
const bytes = Y.encodeStateAsUpdate(doc);
```

如果超过例如 1.5MB，前端提示：

```text
当前文档包含较大图片，已保存在本机，但远端快照可能失败。请压缩或删除部分图片。
```

长期方案再新增附件 API。

## 8. 后续可选的后端附件方案

如果图片演示需要更稳定，建议新增附件存储，而不是把图片原文放进 Yjs。

### 8.1 新增模型

```prisma
model DocAsset {
  groupId         String
  docId           String
  assetId         String
  mime            String
  size            Int
  dataBase64      String   @db.Text
  createdByNodeId String
  createdAt       DateTime @default(now())

  @@id([groupId, docId, assetId])
}
```

### 8.2 新增接口

```text
POST /groups/:id/docs/:docId/assets
GET  /groups/:id/docs/:docId/assets/:assetId
DELETE /groups/:id/docs/:docId/assets/:assetId
```

权限：

- 读取附件需要 doc access >= 1。
- 上传和删除附件需要 doc access >= 2。

安全注意：

- 附件如果明文放后端，会弱化“控制平面不接触内容”的设计。
- 更合适的方式是前端加密附件后上传，后端只保存密文。
- Yjs 中只保存 `assetId`、mime、尺寸、hash 等元数据。

第一阶段为演示方便可以不做后端附件 API。

## 9. 兼容策略

### 9.1 老文档

老文档只有 `Y.Text("content")`，没有 `meta` 和 `blocks`。

加载时：

```ts
const mode = ymeta.get("editorMode") ?? "plain";
```

显示为纯文本模式。

### 9.2 模式切换

模式切换建议规则：

- `plain -> markdown`：保留 `content`，只改变 `meta.editorMode`。
- `markdown -> plain`：仍保留 Markdown 源码，不做破坏性转换。
- `plain/markdown -> blocks`：可以初始化一个 paragraph 或 markdown block，并保留 `content` 作为 fallback。
- `blocks -> markdown/plain`：不自动覆盖 `content`，除非用户明确执行“导出为 Markdown”。

这样可以降低数据丢失风险。

## 10. 实施步骤

### 阶段一：Markdown 演示

目标：最快看到非纯文本效果。

改动范围：

- 新增 `MarkdownSurface`。
- 在 `DocInlineEditor` 中增加模式状态和工具栏。
- 继续使用 `Y.Text("content")`。
- 引入 `react-markdown` 和 `remark-gfm`。

验收：

- 可输入 Markdown。
- 可切换编辑/预览。
- 标题、列表、代码块、表格、任务列表能正确展示。
- 只读用户不能编辑，但能预览。
- P2P 同步 Markdown 源码。

### 阶段二：图片演示

目标：支持插入图片并随文档同步。

改动范围：

- 新增 `yassets` 数据结构。
- 新增 `yblocks` 中的 image block。
- 新增 `BlockDocumentSurface` 或先在 Markdown 模式中支持图片附件。
- 实现图片选择、校验、压缩、插入、删除。

验收：

- 可插入小图片。
- 图片能在另一浏览器实例同步显示。
- 超大图片会被拒绝或压缩。
- 删除图片块后 UI 和 Yjs 状态更新。
- 快照保存不超过后端限制。

### 阶段三：块文档演示

目标：展示多类型内容混排。

改动范围：

- 支持 paragraph、markdown、image、code、todo block。
- 提供添加、删除、上下移动块。
- 将 `DocInlineEditor` 拆为 session 容器和 surface 组件。

验收：

- 一个文档内可混排文本、Markdown、图片、代码块、任务项。
- 各块编辑能通过 Yjs 同步。
- 只读用户不能修改块内容和顺序。

### 阶段四：长期附件存储

目标：支持更大的图片和附件。

改动范围：

- 新增后端附件模型和 API。
- 前端上传加密附件或密文。
- Yjs 仅保存附件引用。

验收：

- 大图片不会撑爆 Yjs snapshot。
- 附件读写仍受文档 ACL 控制。
- 权限撤销和密钥轮换策略有明确处理。

## 12. 风险与处理

| 风险 | 说明 | 处理 |
|------|------|------|
| 图片导致 Yjs update 过大 | base64 图片会显著放大状态体积 | 限制大小、压缩、长期改附件 API |
| Markdown XSS | 如果允许原始 HTML，会有脚本注入风险 | 不启用 raw HTML；只用安全 Markdown 渲染 |
| 多人同时编辑 block 冲突 | 普通 JSON block 整体替换会让并发编辑粒度较粗 | 初期接受；后续将 block 内容改为 Y.Map/Y.Text |
| 组件复杂度上升 | 单个 `DocInlineEditor` 会过大 | 拆分 session 容器和 surface 组件 |
| 老文档兼容 | 旧文档没有 meta/blocks | 默认按 `plain` 加载 |
| 快照保存失败 | 后端限制 2MB | 前端提前检测并提示 |

## 13. 推荐落地顺序

推荐先做：

1. Markdown 编辑/预览。
2. 小图片插入，带大小限制。
3. 简单块文档混排。

原因：

- Markdown 复用现有 `Y.Text("content")`，改动最小。
- 图片是最直观的演示增强，但要控制大小。
- 块文档能统一后续表格、代码、任务项等数据类型。

最终建议架构是：

```text
DocInlineEditor
  ├─ useSecureDocSession / session setup
  ├─ EditorToolbar
  ├─ PlainTextSurface
  ├─ MarkdownSurface
  └─ BlockDocumentSurface
       ├─ ParagraphBlock
       ├─ MarkdownBlock
       ├─ ImageBlock
       ├─ CodeBlock
       └─ TodoBlock
```

这样可以在不破坏现有安全同步链路的前提下，把文档展示能力从纯文本扩展到更适合演示的多数据格式。
